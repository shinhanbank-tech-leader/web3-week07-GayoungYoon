# 스마트컨트랙트 업그레이드 방식과 은행 시스템 변경 방식 비교 분석

- 부서명:  투자서비스개발부 
- 이름:  윤가영

---

스마트컨트랙트는 기본적으로 **배포 후 코드 변경이 어려운 구조**이다. 블록체인에 한 번 올라간 코드는 동일 주소에서 사실상 수정할 수 없고, 이 불변성이 위변조 방지와 신뢰 확보의 핵심이 된다. 그러나 금융 서비스 관점에서는 이 특성이 곧 운영 리스크가 된다. 버그가 발생하거나 정책이 바뀌어도 즉시 수정하기 어렵기 때문이다. 이러한 한계를 보완하기 위해 등장한 방식이 **delegateCall 기반 업그레이드**와 **Proxy 패턴**이다.

delegateCall 방식은 Proxy가 자신의 저장소(storage)는 그대로 유지한 채, 외부 Logic 컨트랙트의 코드를 대신 실행하는 구조이다. 즉 **주소와 데이터는 유지하면서 로직만 교체**하려는 목적에서 나왔다. Proxy 패턴 역시 같은 문제를 해결하기 위해 등장했다. 스마트컨트랙트는 주소가 곧 서비스 진입점이므로, 새 컨트랙트를 다시 배포하면 사용자 연결, 잔액, 상태값 승계가 모두 문제된다. 그래서 로직과 데이터를 분리해 업그레이드 가능성을 확보한 것이다.

하지만 이 구조는 새로운 운영 문제를 만든다. 대표적으로 storage layout 충돌이 발생하면 기존 데이터가 깨질 수 있고, 업그레이드 권한을 가진 관리자 키가 탈취되면 전체 서비스가 공격에 노출될 수 있다. 또한 실제 실행 로직이 분리되어 있어 검증 난이도도 높아진다. 즉 스마트컨트랙트는 “업그레이드가 어려워서” 우회 구조를 만들었지만, 그 대가로 **복잡성과 보안 위험**을 떠안게 된다.

반면 은행 시스템은 처음부터 **변경을 전제로 설계**된다. 예를 들어 코어 계정계 로직 변경, 배치 프로그램 수정, 수수료 정책 반영, ISA 입출금 프로세스 개선 같은 작업은 모두 개발·테스트·승인·배포 절차 안에서 수행된다. 일반적으로 업무변경정의서 작성, 영향도 분석, 개발, 단위테스트, 통합테스트, 운영 승인, 배포, 사후 점검 순으로 진행되며, 필요한 경우 즉시 롤백도 가능하다. 은행 IT에서 중요한 것은 “변경 불가”가 아니라 **변경 가능성을 통제하는 것**이다.

이 점이 두 구조의 가장 큰 차이이다. 스마트컨트랙트는 코드 자체의 불변성으로 신뢰를 확보하려 하지만, 은행 시스템은 승인, 검증, 모니터링, 롤백 같은 **운영 통제 체계**로 신뢰를 확보한다. 은행에서는 실제 운영 반영 전후로 다수의 승인 절차와 검증이 수행되며, 장애 시에는 원복 계획까지 사전에 준비한다. 예를 들어 배치 수정이라면 테스트 데이터 검증, 전후 결과 비교, 운영 반영 승인, 실패 시 롤백 절차가 함께 따라온다. 이는 금융 서비스가 “정확히 동작하는 것”만큼이나 “문제 발생 시 복구 가능한 것”이 중요하기 때문이다.

결국 스마트컨트랙트와 은행 시스템의 차이는 기술 스택의 차이보다 **운영 철학의 차이**에 가깝다. 스마트컨트랙트는 변경을 어렵게 만들어 신뢰를 얻고, 은행 시스템은 변경을 허용하되 강한 통제로 신뢰를 만든다. 운영·통제 관점에서 보면, 스마트컨트랙트는 한 번의 실수가 치명적일 수 있는 구조이고, 은행 시스템은 실수를 전제로 복구와 책임 추적이 가능하도록 설계된 구조라고 볼 수 있다. 따라서 금융 실무에서는 단순히 “업그레이드 가능 여부”보다, **누가 어떤 절차로 변경하고 문제가 생기면 어떻게 통제·복구할 수 있는가**가 더 본질적인 차이라고 할 수 있다.
